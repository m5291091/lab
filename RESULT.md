## 1. 実験環境と計測方法

### ハードウェア仕様

| 項目 | 仕様 |
|------|------|
| システム | Miyabi-G (NVIDIA GH200 Grace Hopper Superchip × 1 node) |
| CPU | NVIDIA Grace (72-core Arm Neoverse V2) |
| CPU メモリ | 480 GB LPDDR5X |
| GPU | NVIDIA Hopper H100 (132 SM, CUDA Compute 9.0) |
| GPU メモリ | 96 GB HBM3 |
| CPU↔GPU 接続 | NVLink-C2C (双方向 900 GB/s 理論値) |
| CUDA バージョン | 12.x |
| コンパイラ | NVCC + GCC (Release ビルド, `-O3 -arch=sm_90`) |

### 計測方法

- 各実装を **3 回以上実行**し、中央値を採用
- `Sequential` と `OpenMP` は CPU 計測 (時間は `omp_get_wtime()`)
- GPU 実装は `cudaEventRecord` で GPU カーネル全体の時間を計測
- GTEPS = (全ソース数 × 辺数 × 2) / 実行時間 [Giga Traversed Edges Per Second]
- バッチサイズ: GPU 実装は頂点数に応じて自動選択 (デフォルト 32〜256)

## 2. メモリ帯域計測

`bandwidth_benchmark` による GH200 の実測メモリ帯域。  
NVLink-C2C 帯域が HBM3 の約 1/9.6 であることが、GPU_Managed 実装の性能低下の直接原因。

| 転送種別 | 転送サイズ 0.1 GB | 0.5 GB | 1.1 GB | 4.3 GB | **代表値** |
|---------|:---:|:---:|:---:|:---:|:---:|
| HBM3 D→D (デバイス内コピー) | 1427 | 1464 | 1477 | **1488** | **1488 GB/s** |
| NVLink-C2C (CPU→GPU 方向) | 152 | 154 | 175 | **156** | **156 GB/s** |
| Pinned D→H (GPU→ホスト) | 294 | 297 | 298 | 298 | **298 GB/s** |
| Pinned H→D (ホスト→GPU) | 410 | 421 | 423 | **425** | **425 GB/s** |

**HBM3 vs NVLink-C2C の比: 9.6 倍**

```
HBM3 D→D    : ████████████████████████████████████████  1488 GB/s (理論値 4020 の 37%)
NVLink-C2C  : ████                                       156 GB/s  (理論値 900 の 17%)
Pinned H→D  : ████████████                               425 GB/s
Pinned D→H  : ████████                                   298 GB/s
```

> **考察**: GPU_Managed 実装は、デフォルトでグラフ構造データが CPU 側の LPDDR5X に配置されるため、
> GPU からのアクセスは NVLink-C2C 経由となる。BC 計算はグラフトポロジへの大量ランダムアクセスが発生するため、
> HBM3 比 9.6 倍遅い NVLink-C2C がボトルネックとなる。
> 手法 1 (`SetReadMostly + PrefetchAsync`) によりデータを HBM3 にキャッシュさせることで
> このボトルネックを解消する。

## 7. アブレーションスタディ (vs GPU ベースライン)

4 グラフで GPU (`cudaMalloc` + HBM3 直接確保) を 1.0 とした場合の相対性能。  
各手法の独立した貢献度を定量化する。

| グラフ | 規模 | GPU (= 1.0) | GPU_Managed | GPU_ReadMostly | **GPU_Opt** |
|--------|------|:-----------:|:-----------:|:--------------:|:-----------:|
| benchmark_7000 | 7K | 1.000 | **1.183** | 1.439 | 1.416 |
| benchmark_11023 | 11K | 1.000 | 0.710 | 0.982 | 0.982 |
| 56438 | 56K | 1.000 | 0.196 | 1.000 | **1.132** |
| 325557 | 325K | 1.000 | 0.198 | 0.999 | **1.062** |

*値が大きいほど速い (= GPU ベースライン比でのスピードアップ)*

### アブレーション解釈

| 比較 | 定量 | 解釈 |
|------|------|------|
| GPU → GPU_Managed | 56K: **0.20**, 325K: **0.20** | NVLink-C2C ボトルネックにより 5 倍以上の性能低下 |
| GPU_Managed → GPU_ReadMostly | 56K: 0.196 → **1.000**, 325K: 0.198 → **0.999** | **手法 1** (ReadMostly + Prefetch) が完全に性能を回復 |
| GPU_ReadMostly → GPU_Opt | 56K: 1.000 → **1.132**, 325K: 0.999 → **1.062** | **手法 2** (2-stream 非同期) でさらに 6〜13% 高速化 |
| GPU → GPU_Opt | 56K: 1.000 → **1.132**, 325K: 1.000 → **1.062** | 提案手法全体で GPU ベースライン超過 |

> **手法 1 の効果 (小グラフ)**: 7K グラフでは GPU_Managed が GPU より速い (1.183×)。  
> これは 7K グラフが HBM3 には収まらず LPDDR5X に配置されるが、  
> グラフが小さいためページフォルトのオーバーヘッドが相対的に低く、  
> HBM3 の初期化コストが節約できるためと考えられる。

```
アブレーション可視化 (GTEPS, 4グラフ):

  benchmark_7000_41459  : GPU=0.95, Managed=1.12, ReadMostly=1.36, Opt=1.34
  benchmark_11023_62184 : GPU=2.35, Managed=1.67, ReadMostly=2.30, Opt=2.30
  56438_300801          : GPU=9.85, Managed=1.93, ReadMostly=9.86, Opt=11.16 ★
  325557_3216152        : GPU=8.31, Managed=1.64, ReadMostly=8.31, Opt=8.83  ★

  ★ 大規模グラフで手法 1+2 の両方の効果が顕著
```

## 8. 主要な知見と考察

### 知見 1: NVLink-C2C ボトルネックの定量化

- 実測 NVLink-C2C 帯域: **156 GB/s**、HBM3 帯域: **1488 GB/s** → **比率 9.6 倍**
- GPU_Managed では中規模グラフで GPU 比 **4.9〜5.1 倍の性能低下**
- これは帯域比 (9.6 倍) の約 1/2 程度であり、BC のメモリアクセスパターンが  
  完全ランダムではなく BFS の局所性によって部分的に緩和されることを示す

### 知見 2: ReadMostly 手法の有効性

- `GPU_ReadMostly` は **全グラフで GPU ベースライン性能をほぼ完全回復** (99〜100%)
- 「読み取り専用」ヒントにより HBM3 の L2 キャッシュに CSR データの重複コピーが保持される
- 同一ソース頂点バッチ内で隣接リストへの繰り返しアクセスがあるため、  
  キャッシュヒット率が上昇し NVLink-C2C トラフィックを削減

### 知見 3: 2-stream パイプラインの効果

- `GPU_Opt` は `GPU_ReadMostly` 比で **6〜13% の追加高速化** (中〜大規模グラフ)
- GH200 の Copy Engine が SM とは独立して動作するため、  
  `cudaMemsetAsync` による BFS バッファ初期化とカーネル実行が真に重複実行される
- 小規模グラフ (7K, 11K) では `cudaMemsetAsync` のコストが支配的ではないため効果は小さい

### 知見 4: グラフ種別による傾向の違い

| グラフ種 | GPU_Opt の傾向 | 理由 |
|---------|-------------|------|
| web グラフ (NotreDame, Stanford) | 高 GTEPS (10〜44) | べき乗則分布、高次数頂点が多い → BFS が広く伝播 → 高並列性 |
| road ネットワーク (TX, PA, CA) | 低 GTEPS (0.4〜0.5) | 格子状、平均次数 2〜3 → BFS が狭い → 並列性が低い |
| amazon グラフ | 中 GTEPS (3.7〜5.0) | 共購買ネットワーク、中程度の次数分布 |

### 知見 5: 実用的指針

- グラフが **HBM3 の 35% 以内** (≈ 33 GB) に収まる場合は `GPU` (cudaMalloc) が最速
- 大規模グラフ (> 33 GB) でも **手法 1+2** (`GPU_Opt`) が自動的に最適な配置を選択
- OpenMP との交差点は約 **56K ノード以上** で GPU が優位
